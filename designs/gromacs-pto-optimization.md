# Gromacs + PTO 优化加速方案设计

## 1. 优化概述

基于前面对PTO机制和Gromacs计算热点的分析，本方案提出将Gromacs核心计算热点通过PTO编程范式进行重构，利用PTO的Tile-based分层抽象和算子融合能力，在昇腾AI加速器上获得更好性能。

## 2. 整体设计思路

### 2.1 设计原则

1. **热点优先**：优先优化占比最大的非键相互作用和PME计算
2. **增量优化**：保持Gromacs整体框架不变，只替换核心计算kernel
3. **分层优化**：利用PTO多层次IR，从Tensor级自动融合开始，逐步深入Tile级手工优化
4. **保持精度**：优化不改变计算结果和物理精度

### 2.2 总体架构

```
┌─────────────────────────────────────────────┐
│  Gromacs Host 框架 (mdrun, I/O, 调度)       │
├─────────────────────────────────────────────┤
│  ↓ 热点计算offload                           │
├─────────────────────────────────────────────┤
│  PyPTO 编程框架                              │
│  ┌─────────┐ ┌─────────┐ ┌───────────────┐  │
│  │非键交互 │ │PME计算  │ │ 约束算法      │  │
│  │ 融合   │ │ 融合    │ │  融合         │  │
│  └─────────┘ └─────────┘ └───────────────┘  │
├─────────────────────────────────────────────┤
│  PTO ISA 虚拟指令集                          │
├─────────────────────────────────────────────┤
│  昇腾AI加速器硬件                            │
└─────────────────────────────────────────────┘
```

## 3. 具体优化方案

### 3.1 方案一：非键相互作用算子融合优化

#### 3.1.1 当前问题分析

当前GPU实现通常：
- 邻居列表构建 → 单独kernel
- 范德华力计算 → 单独kernel
- 静电力短程部分 → 单独kernel
- 力累加归约 → 单独kernel

问题：
- 多个kernel启动开销
- 邻居坐标中间结果写回HBM
- 累加过程多次内存读写

#### 3.1.2 PTO融合策略

**垂直融合方案**：将整个非键相互作用计算流程融合为单个PTO kernel：

```
原始：NeighborList → LoadCoords → VdWCalc → Electrostatic → ReduceForce
PTO融合：
  Tile划分：按原子块分Tile
  Tile内：
    1. 加载Tile内原子坐标 (TLOAD)
    2. 计算所有原子对距离
    3. 计算范德华能量和力
    4. 计算短程静电能量和力
    5. 累加力到输出 (TSTORE)
  ↓
  单个PTO kernel完成整个计算
```

**融合策略**：
- **Tile粒度**：每个Tile处理32-64个原子
- **寄存器/SPAM利用**：Tile内坐标保留在SRAM，不写回
- **计算重排**：邻域对计算按Tile分块，提高数据复用

#### 3.1.3 预期收益

- 减少kernel启动次数：N → 1
- 减少全局内存访问：邻居坐标不写回，估计节省 ~20-30% 内存访问
- 计算密度提升：每个Tile内更多计算，更好掩盖访存延迟
- **预期性能提升**：15-25%

### 3.2 方案二：PME全流程融合优化

#### 3.2.1 当前问题分析

当前PME实现通常分多步：
```
Charge Spreading (电荷散布) → kernel 1
3D FFT → 库函数调用
Spectral convolution → kernel 2
Inverse FFT → 库函数调用
Force interpolation → kernel 3
```

问题：
- 五个独立kernel，四次全局内存读写网格数据
- 每个kernel启动开销累积
- 网格数据在HBM中往返传输，带宽浪费

#### 3.2.2 PTO融合策略

**分段融合方案**：

**阶段1：电荷散布融合**
```
AtomCharge → ScatterToGrid → GridBuffer
  ↓ PTO融合
单个Tile操作完成原子电荷到三维网格散射
```

**阶段2：卷积计算融合**
```
After FFT: Grid × GreenFunction → 相乘
  ↓ PTO融合
每个Tile网格点独立计算，融合乘法操作
不需要额外kernel调用
```

**阶段3：力插值融合**
```
GridGradient → GatherToAtomForce
  ↓ PTO融合
单个Tile操作完成网格梯度到原子力收集
```

**整体PME融合后**：
- 电荷散布 → 融合kernel 1
- 3D FFT → 保持库调用（已高度优化）
- 谱卷积 → 融合到PTO，与前/后阶段合并
- 逆FFT → 保持库调用
- 力插值 → 融合kernel 3

#### 3.2.3 进阶融合：全PME流水线融合

对于足够大的SRAM，可以进一步融合：
```
ChargeSpreading + 零拷贝 FFT输入 + 谱卷积 + 逆FFT输出 + ForceInterpolation
        ↓
    整个流程在PTO框架内调度，减少两次HBM往返
```

#### 3.2.4 预期收益

- 减少2-4次全局内存写回
- 减少kernel启动开销
- 更好利用昇腾向量单元进行网格计算
- **预期性能提升**：PME阶段 10-20% → 整体性能提升 2-6%

### 3.3 方案三：LINCS约束算法融合优化

#### 3.3.1 当前问题分析

当前GPU实现中：
- 每次迭代调用一个小kernel
- 多次迭代导致多次kernel启动
- 稀疏矩阵向量乘之间数据写回

#### 3.3.2 PTO融合策略

**迭代内融合**：将LINCS的多次迭代融合到单个PTO kernel：
```
Original: A × M → kernel 1
          projection → kernel 2
          iteration 1 → kernel 3
          iteration 2 → kernel 4

PTO融合: 所有迭代在一个Tile内完成
  - 迭代中间结果保留在SRAM
  - 减少HBM访问
```

**按约束链分Tile**：每个PTO Tile处理一条或多条约束链，串行迭代但不离开kernel

#### 3.3.3 预期收益

- 减少kernel启动次数
- 减少中间结果内存访问
- **预期性能提升**：约束阶段 20-30% → 整体性能提升 1-4%

### 3.4 方案四：邻居列表构建融合

#### 3.4.1 PTO融合策略

将网格桶排序 + 距离过滤融合为单个PTO kernel：
- 每个Tile负责一个空间块
- 排序和筛选在Tile内完成
- 直接输出邻居列表到SRAM/HBM

#### 3.4.2 预期收益

整体提升不大，约 **0.5-2%** 整体性能提升。

## 4. 算子融合策略总结

### 4.1 融合策略分类

| 优化对象 | 融合方式 | 融合等级 | 实现难度 |
|----------|---------|---------|---------|
| 非键相互作用 | 垂直全流程融合 | Tile级 | 中等 |
| PME电荷散布+力插值 | 相邻操作垂直融合 | Tile级 | 低 |
| PME谱卷积 | 水平融合 | Tensor级 | 低 |
| LINCS多迭代 | 迭代垂直融合 | Tile级 | 中等 |
| 邻居列表构建 | 排序+过滤融合 | Tensor级 | 低 |

### 4.2 融合优先级推荐

1. **第一优先级**：非键相互作用全融合 → 收益最大
2. **第二优先级**：PME各步骤融合 → 收益第二
3. **第三优先级**：LINCS约束迭代融合 → 收益中等
4. **第四优先级**：邻居列表构建融合 → 收益最小

## 5. 预期性能收益估算

基于各阶段计算占比和预期提速，估算总体性能提升：

| 计算组件 | 原始占比 | 组件内提速 | 整体贡献 |
|----------|----------|-----------|---------|
| 非键相互作用 | 70% | 15-25% | 10.5-17.5% |
| PME | 15% | 10-20% | 1.5-3.0% |
| LINCS约束 | 8% | 20-30% | 1.6-2.4% |
| 邻居列表 | 3% | 15-20% | 0.45-0.6% |
| **总计** | **96%** | - | **14.05-23.5%** |

### 5.1 不同场景下的总体预期

| 场景 | 体系大小 | 预期整体性能提升 |
|------|----------|-----------------|
| 小体系 (<10万原子) | 非键占比更高 | 15-25% |
| 中等体系 (10-100万原子) | 均衡 | 14-22% |
| 大体系 (>100万原子) | PME占比增加 | 12-20% |
| 多节点并行 | 通信占比增加 | 10-18% |

**中心点估计**：在昇腾AI加速器上，通过PTO融合优化，整体性能可望获得 **15-20%** 的提升。

## 6. 可能的挑战和限制

### 6.1 技术挑战

1. **Tile大小选择**：
   - 挑战：需要根据原子密度和截止半径选择合适Tile大小
   - 应对：自动调优+参数化，支持不同体系选择最优值

2. **SRAM容量限制**：
   - 挑战：大规模Tile需要更多SRAM存储坐标和力
   - 应对：分块策略，自适应Tile大小，利用分层存储

3. **负载不均衡**：
   - 挑战：不同空间区域原子密度不同，Tile计算量差异大
   - 应对：动态调度+原子密度感知的Tile划分，PTO MPMD调度天然支持

4. **FFT库集成**：
   - 挑战：PME中FFT需要调用优化库，融合边界设计
   - 应对：只融合相邻算子，保持FFT库调用，数据零拷贝传递

### 6.2 约束条件

1. **硬件平台限制**：仅适用于华为昇腾AI加速器（PTO存在的硬件平台）
2. **编程模型变更**：需要将现有Gromacs CUDA/OpenCL核移植到PTO范式
3. **维护成本**：增加额外代码路径，需要持续维护
4. **精度保证**：融合可能改变计算顺序，需要验证精度正确性

### 6.3 不适合PTO优化的部分

1. **全局通信**：多节点PME通信已经是优化瓶颈，PTO无法解决
2. **I/O和数据输出**：这部分本身不是计算热点
3. **已经高度优化的BLAS/FFT**：保持调用现有优化库更好

## 8. 结论

华为PTO (Parallel Tile Operation) 编程范式非常适合对Gromacs进行优化加速：

1. **技术适配性好**：Gromacs大部分核心计算可以表示为Tile并行张量操作
2. **融合机会丰富**：现有实现多阶段拆分存在大量kernel间数据移动，融合可以显著减少开销
3. **预期收益可观**：整体可望获得15-20%的性能提升，在热点组件上提升更大
4. **实现路径清晰**：增量优化，热点优先，可以逐步落地

PTO基于Tile的分层抽象设计，正好匹配分子动力学模拟中空间局域性计算特点，有望在昇腾平台上充分发挥硬件潜力，获得显著性能提升。
